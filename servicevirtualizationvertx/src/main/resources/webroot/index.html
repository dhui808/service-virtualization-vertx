Develop Modern Web Applications with Service Virtualization

In a nutshell, service virtualization provides a set of mocking services that simulate the real services.

Why bother with service virtualization?

Because the real services are not always available. This is becoming even more pronounced since modern applications adopt increasingly distributed architecture via microservices. As such, developers and QA frequently find themselves unable to do testing since a backend service is down. Their productivity is hit hard and the companies find their IT cost soaring.
Even if the real services are available, it is difficult or impossible to test the application behaviour in a particular error scenario. For example, if we need to test how the UI behaves when a backend service got a problem with its dependency, how can the backend service achieve this condition?

The service virtualization implemented in this PoC project has a unique feature that no other vendor products offer: it allows the multiple different responses for the same request. This is exactly what the UI developers need.

How is this achieved?

It is actually simple and intuitive - the UI developer has to select which scenario he/she wants to test, before doing the test.

In this project, the four test scenarios in a login flow for a banking web application are included.

Test Steps:

Step 1: Click one of the scenario you want to test (i.e. you need to go to the desired page so you can continue developing it or fix defect.)

Step 2: The login page will open in a new tab (please allow pop-ups if your browser blocks them.). If it doesn't, click here (please enable pop-ups in you browser setting)., or manually open the URL in a new tab:

Step 3: Log in with any card number and password, or just leave them blank

Test Scenarios:

  1. Login successful, no MFA, Account Detail successful
  2. Login failed
  3. Login successful, no MFA, Account Details failed
  4. Login successful, with MFA, Account Detail successful
